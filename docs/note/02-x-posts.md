# X集客投稿(10日分)

note本文(`01-article-draft.md`)から逆算した投稿計画。
noteが先、投稿があと。投稿の主張はすべてnote無料部分と一致させてある。

## 運用ルール(垢BAN対策・必読)

- **投稿はすべて手動で行う。** 自動投稿ツール・予約の一括連投は使わない(Xの自動化ポリシー抵触リスク)
- **リンク付き投稿は10本中3本まで**(Day 3, 6, 10)。残りは誘導なし〜弱
- 同一文面・ほぼ同一文面の再投稿はしない。過去投稿のコピペ連投はスパム判定の典型例
- 数字・実績は実測できるものだけ(「5分ごと」「0円」はコードで裏取り済み)。収益・販売部数の誇張はしない
- リプ欄で他人の投稿に無差別にnoteリンクを貼らない
- 1日1本。伸びなくても頻度を上げて連投しない

### Day 1|悩み共感(誘導なし)

> また売り切れてた。
>
> Apple整備済品、朝はあったのに昼には消えてる。
>
> 「こまめにチェックしよう」って思うんだけど、あれ続かないんですよね。
>
> 見るたびに「ない」を確認する作業、地味にしんどい。

狙い:ターゲット(整備済品を狙っている人)の共感リプ・いいねで、この悩みを持つフォロワーを可視化する。売り込み要素ゼロ。

### Day 2|勘違いを壊す(誘導なし)

> 整備済品が買えないのは、チェックの回数が足りないからじゃないです。
>
> あれは「見た時に買う」ゲームじゃなくて「出た瞬間に知る」ゲーム。
>
> 人間が1日10回見ても、監視できるのは10個の「点」だけ。
>
> 在庫は点と点の間に出て、消えます。

狙い:note無料部分の核となる主張の切り出し。「たしかに」と思わせて翌日の告知の下地を作る。

### Day 3|note公開告知(誘導:強)★リンクあり

> 欲しいMacが整備済品に出た瞬間、iPhoneに通知が来る仕組みを作りました。
>
> ・サーバー代0円(GitHub Actionsだけ)
> ・専用アプリ不要(GitHub公式アプリが通知してくれる)
> ・検出から通知まで最大5〜7分
>
> 仕組みの全体像は無料部分に全部書きました。
> 手順とコード全文は有料部分にあります。
>
> [noteリンク]

狙い:公開告知。便益(通知が来る)+コスト(0円)+無料部分だけでも価値がある、を明示。「絶対買える」等の保証表現は使わない。

### Day 4|主張の切り出し(誘導:弱、リンクなし)

> 「見に行く」のをやめて「来させる」。
>
> 在庫チェックって、botに任せられる仕事の代表例だと思う。
>
> 5分ごとにストアを見に行って、新着があった時だけ僕を呼ぶ。
> 人間がやるのは、通知が来てから買うかどうかの判断だけ。
>
> 詳しくは昨日のnoteに書きました。

狙い:noteの設計思想を1ツイートに圧縮。リンクは貼らず「昨日のnote」で言及に留め、プロフィール/固定ポスト経由の導線にする。

### Day 5|技術的な裏側(誘導なし)

> 在庫監視botの構成、全部GitHubに寄せると気持ちいいですよ。
>
> ・実行環境 → GitHub Actions(Publicリポジトリなら無料)
> ・通知 → Issue作成 = GitHub Mobileがプッシュ通知
> ・DB → JSONファイル1個をbotが自分でcommit
>
> 外部サービス登録、ゼロ。

狙い:エンジニア層・学習者層へのリーチ。RTされやすい「構成ネタ」。売り込みなしで信頼を積む。

### Day 6|失敗談→解決(誘導:強)★リンクあり

> 【要追記:実際に買い逃した体験を1行で。例「◯◯を買い逃したのが悔しくて監視botを作った」】
>
> 悔しかったので、在庫チェックを自動化しました。
>
> 作ってみたら、必要だったのはサーバーでも月額サービスでもなく、GitHubアカウント1個だけでした。
>
> 作り方は全部ここに:[noteリンク]

狙い:感情(悔しさ)から入る導線。Day 3と文面・切り口を変えた2度目のリンク投稿。

### Day 7|ハウツーの無料開示(誘導なし)

> GitHubのIssueは「無料で使えるプッシュ通知API」として優秀です。
>
> 1. 公式アプリを入れる
> 2. リポジトリをWatch → Custom → Issues
> 3. あとはスクリプトから issue create するだけ
>
> LINE APIの登録もSlackのWebhookも要らない。個人の通知はこれで足りる。

狙い:単体で役に立つTipsを無料で配る。「この人の情報は具体的」という信頼を作る。noteの有料部分の核心(全手順・カスタマイズ)は明かさない。

### Day 8|Q&A(誘導:弱、リンクなし)

> noteに「プログラミング未経験でも作れますか?」と質問をもらったので答えると:
>
> コマンドは全部コピペで動くように書きました。
> 必要なのは経験より「エラーが出たら手順を読み直す」根気です。
>
> そこまで含めて、つまずきポイント集を有料部分に入れてあります。

狙い:購入を迷っている層の不安(自分にできるか)を解消する。※実際に質問が来ていない場合はこの投稿はスキップするか、「よく聞かれそうなこと」と正直な表現に変える。**架空の質問をでっち上げない。**

### Day 9|別角度の便益(誘導なし)

> 在庫監視botの副産物:ストアを見に行く回数がゼロになったこと。
>
> 「そろそろ出たかな」って頭の片隅で考え続けるの、地味に脳のリソース食ってたんだなと。
>
> 通知が来るまで忘れていられるのが、いちばんの価値かもしれない。

狙い:機能(通知)ではなく情緒(気にしなくていい)の便益。共感層を広げる。

### Day 10|まとめ+再告知(誘導:強)★リンクあり

> 今週書いたnoteのまとめ。
>
> Apple整備済品を買い逃し続けた僕が、
> ・0円で(GitHub Actions)
> ・5分間隔の在庫監視を作り
> ・出た瞬間iPhoneに通知が来るようにした
> 話です。
>
> 無料部分だけで仕組みは全部わかります。再現手順が欲しい人だけ有料へ。
>
> [noteリンク]

狙い:1週間の集大成。「無料部分だけで仕組みは全部わかる」と明言することで、誠実さを担保しつつ最後の誘導をする。
