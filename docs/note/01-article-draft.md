# (ここから本文)

Apple整備済品の在庫チェック、もうやめました。

代わりに、欲しいMacが出た瞬間にiPhoneへ通知が来るようにしました。

サーバー代は0円です。

このnoteでは、その仕組みの全体像と、実際に動いているコードの話をします。

有料部分では、あなたのGitHubアカウントで今日から動かすための全手順と、コード全文を渡します。

## 整備済品が買えないのは、運じゃない

Apple整備済品は、新品より安くて、バッテリーも外装も新品同様で、保証も同じ。

だから人気モデルは一瞬で消えます。

【要追記:あなたの体験。例:どのモデルを、いつ、どう買い逃したか。朝見たら消えていた、など具体的な1シーン】

ここで多くの人は「こまめにチェックしよう」と考えます。

僕もそうでした。

でも、これは根性の問題ではなく、構造の問題です。

## 手動リロードでは構造的に勝てない

整備済品は「見た時に買う」ゲームではありません。

「出た瞬間に知る」ゲームです。

人間が1日に10回ストアを開いても、監視できるのは10回分の「点」だけ。

在庫が出てから消えるまでの時間が短いモデルでは、点と点の間にすべてが終わります。

しかも、このチェック習慣は続きません。

疲れるし、見るたびに「ない」を確認する作業はしんどい。

だから発想を変えます。

**「見に行く」のをやめて、「来させる」。**

## 仕組みの全体像

作ったのは、こういうシステムです。

```
GitHub Actions (5分ごとのcron)
  └─ Apple整備済ストアをチェック
  └─ 前回との差分を計算
  └─ 新着があれば GitHub Issue を作成
        └─ GitHub Mobile アプリが iPhone にプッシュ通知
```

流れは4ステップ。

1. GitHubの自動実行機能(Actions)が5分おきにスクリプトを起動する
2. スクリプトがAppleの整備済ストアのページから商品一覧を取り出す
3. 「前回見た商品リスト」と比べて、新しく増えた商品だけを見つける
4. 新着があったら、自分のリポジトリにIssue(掲示板の投稿のようなもの)を1件作る

ポイントは4つ目です。

GitHubの公式アプリ(GitHub Mobile)は、Issueが作られるとプッシュ通知を出せます。

つまり、**Issue作成 = iPhoneのロック画面に通知**。

通知をタップすると、Issueに貼られたAppleの商品ページへ直接飛べます。

検出から通知までは、cronの遅延を入れても最大5〜7分ほどです。

売り切れ(商品の消失)は通知しません。買えない情報が来ても疲れるだけなので。

## なぜこの構成にしたか

似たものを作ろうとして、多くの人が止まるのはここです。

「監視スクリプトはどこで動かし続けるの?」

自宅PCの常時起動はうるさいし、クラウドサーバーは月額がかかる。

そこで全部GitHubに寄せました。

- **実行環境:** GitHub Actions。Publicリポジトリなら無料で使える
- **通知:** GitHub Issue + 公式モバイルアプリ。LINE APIの登録も、Slackの設定も、専用アプリも不要
- **データベース:** なし。「前回見た商品リスト」はJSONファイル1個で、botが自分でリポジトリにコミットして保存する

サーバー0円、通知0円、DB0円。

登録が必要な外部サービスはGitHubだけです。

技術的な中身も、実は硬派です。

- 商品一覧は、ページ内に埋まっている `"tiles":[...]` というデータの塊を取り出して読む
- 4つのURLを並列で取りに行って、商品番号(partNumber)で重複を除く
- アクセスは5分に1回・数リクエストだけ。相手のサーバーに負荷をかけない頻度に抑えている

そして白状すると、このコードの大部分はClaude Code(AIコーディングエージェント)に書いてもらいました。

## 1. 必要なもの

- GitHubアカウント(無料)
- iPhoneまたはAndroid(GitHub Mobileアプリを入れる)
- PCのターミナルを開くことへの抵抗がないこと(コマンドはすべてコピペ可)

プログラミング経験は必須ではありません。

ただし「エラーが出たら該当セクションを読んで対処する」姿勢は必要です。

## 2. セットアップ全手順

### 2-1. リポジトリを作る

Publicリポジトリとして作成します(Publicだと Actions が無料で使えるため)。

```bash
git init -b main
git add .
git commit -m "init: apple refurbished watcher"
git remote add origin git@github.com:<あなたのユーザー名>/apple-refurb-watch.git
git push -u origin main
```

### 2-2. Actions の権限設定

リポジトリの Settings → Actions → General で:

- Workflow permissions を **Read and write permissions** に変更

これを忘れると、Issue作成と状態ファイルの保存が両方失敗します(ハマりどころ第1位)。

### 2-3. 通知用ラベルを作る(任意)

```bash
gh label create refurb-new --color "00ff00" --description "Apple refurb new arrival"
```

作らなくても動きます。ラベルが無い場合は、ラベルなしでIssueを作るフォールバックが入っています。

### 2-4. iPhone側の設定

1. App Storeから **GitHub**(公式アプリ)をインストール
2. サインインして Settings → Notifications → プッシュ通知をオン
3. リポジトリページ → Watch → **Custom → Issues** にチェック

これで「Issueが作られた瞬間にロック画面に通知」が完成します。

### 2-5. 動作テスト

```bash
gh workflow run watch.yml --field iterations=1 --field interval=1
gh run list --workflow watch.yml --limit 1
```

ログに `total=22 watched=10 known=16 new=0` のような1行が出れば正常です。

`new=0` は「新着なし」の意味で、エラーではありません。

## 3. コード全文と解説

ファイルは実質3つだけです。

```
├── .github/workflows/watch.yml   # 5分cronでスクリプトを起動
├── scripts/
│   ├── parser.py                 # HTML → 商品リスト
│   └── check.py                  # 取得 + 差分検出 + Issue作成 + 状態保存
└── state/known_products.json     # 既知の商品番号リスト(botが自動更新)
```

### watch.yml(自動実行の設定)

```yaml
on:
  schedule:
    - cron: '*/5 * * * *'   # 5分ごと(Actionsの実質的な最短間隔)
concurrency:
  group: refurb-watch        # 前の実行が終わる前に次が始まらないようにする
  cancel-in-progress: false
permissions:
  contents: write   # 状態ファイルのコミットに必要
  issues: write     # Issue作成に必要
```

【公開時にコード全文を貼る:.github/workflows/watch.yml】

見どころは3つ。

- `concurrency` で多重起動を防ぐ(重複通知の予防)
- `timeout-minutes: 3` で、固まっても3分で強制終了
- 最後のステップで、更新された商品リスト(JSON)をbot自身がコミットする

### parser.py(商品一覧の取り出し)

Appleのページには、商品データが `"tiles":[...]` というJSON配列の形で埋まっています。

このパーサーは、`"tiles":[` を見つけたら、`[` と `]` の対応を1文字ずつ数えて配列の終わりを特定します。

文字列の中にある括弧やエスケープも考慮しているので、HTMLの見た目が多少変わっても壊れにくい。

HTMLパーサーのライブラリ(BeautifulSoup等)を使わないのは、依存を減らすためと、この用途では括弧の対応を数える方が確実だからです。

【公開時にコード全文を貼る:scripts/parser.py】

### check.py(本体)

やっていることは4つです。

1. 4つのURL(全体+カテゴリ別フィルタ3つ)を **aiohttpで並列取得**し、partNumberで重複排除
2. 監視ルールに合う商品だけに絞る
3. `known_products.json` の既知リストと差分を取り、増えた商品だけ抽出
4. 新着があれば `gh issue create` でIssueを作成

【公開時にコード全文を貼る:scripts/check.py】

細かい工夫:

- 4URLのうち一部が失敗しても続行し、**全部失敗した時だけ**エラーにする
- `GITHUB_REPOSITORY` 環境変数が無いローカル実行では、Issueを作らず内容を表示するだけ(dry-run)になる。手元で安全にテストできる
- ブラウザ相当の User-Agent と `Accept-Language: ja-JP` を付けて、日本ストアのページを確実に取る

## 4. カスタマイズレシピ:監視条件を自分仕様にする

監視ルールは `parser.py` の `is_watched()` に集約してあります。

初期設定はこうです(僕の欲しいものリストそのまま):

```python
WATCHED_CATEGORIES = {"macstudio", "macmini", "macbookpro"}

def is_watched(product: Product) -> bool:
    if product.category not in WATCHED_CATEGORIES:
        return False
    if product.category == "macbookpro":
        return normalize_memory_gb(product.memory) >= 128  # MBPは128GBのみ
    return True
```

### レシピ1:MacBook Pro を64GB以上に緩める

```python
    if product.category == "macbookpro":
        return normalize_memory_gb(product.memory) >= 64
```

### レシピ2:iMacも監視対象に足す

```python
WATCHED_CATEGORIES = {"macstudio", "macmini", "macbookpro", "imac"}
```

カテゴリ名は、Appleのフィルタ値(`refurbClearModel`)と同じ表記を使います。

### レシピ3:価格の上限で絞る

```python
    if product.price_raw > 300000:  # 30万円超は通知しない
        return False
```

`price_raw` には税込価格が数値で入っています。

### レシピ4:通知の頻度を下げる

5分おきが多すぎると感じたら、`watch.yml` の cron を `*/10 * * * *`(10分おき)に変えるだけです。

## 5. つまずきポイント集

**通知が来ない**
→ ほとんどの場合、監視対象の在庫がそもそもゼロです。Actionsのログで `watched=0` を確認。仕組みは正常です。

**`gh issue create` がエラーになる**
→ Settings → Actions の Workflow permissions が Read only のまま。2-2をやり直す。

**同じ商品の通知が2回来る**
→ 状態ファイルのコミットが失敗している可能性。Actionsログの「Commit state changes」ステップを確認。push失敗は次回実行で自動リカバリされる設計ですが、連続して失敗する場合は権限を疑う。

**Actionsが動かない**
→ Privateリポジトリだと無料枠(月2,000分)を使い切ると止まります。Publicにするのが確実。

**ある日突然、商品が取れなくなった**
→ Appleがページ構造を変えた可能性。`"tiles":[` がページ内にあるかを確認。この文字列ごと変わっていたら `parser.py` の `marker` を新しい構造に合わせる。

## 6. 応用:Apple以外に転用する考え方

この仕組みの本質は「**一覧ページを定期取得 → 前回との差分 → Issueで通知**」で、Apple専用なのは `parser.py` だけです。

転用の手順:

1. 対象サイトの商品一覧ページに、データがJSONで埋まっていないか探す(ブラウザの開発者ツールでページソースを検索)
2. あれば `extract_tiles()` の `marker` を差し替える。なければHTMLパースに書き換える(ここはClaude Codeに「このHTMLから商品名と価格の一覧を抜くパーサーを書いて」と頼むのが早い)
3. `is_watched()` を自分の条件に書き換える

**ただし、必ず守ること:**

- 対象サイトの利用規約を確認する(スクレイピングを明示的に禁止しているサイトには使わない)
- アクセス頻度は控えめに(このシステムは5分に1回・数リクエスト。それ以上攻めない)
- 転売目的の買い占め自動化には使わない。これは「自分が欲しいものを買い逃さない」ための道具です
